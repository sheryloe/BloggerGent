# Step 7. 게시 안전장치와 SEO 메타 검증을 왜 같이 묶었는가

## 추천 제목

1. 공개 글 덮어쓰기 방지와 SEO 메타 검증을 같이 설계한 이유
2. Blogger 운영에서 가장 위험한 두 가지: 잘못된 업데이트와 잘못된 메타데이터
3. Blogger SEO는 왜 DB 값이 아니라 공개 페이지 기준으로 봐야 하는가

## 검색 설명

공개 글 덮어쓰기 방지, 중복 주제 차단, SEO 메타 검증을 한 번에 설계해야 운영 사고를 줄일 수 있다. BloggerGent에서 적용한 기준을 정리했다.

## 추천 태그

## 본문

블로그 운영에서 가장 무서운 순간은 글이 안 써지는 순간이 아니라, 이미 잘 돌아가던 공개 글이 잘못 수정되는 순간이다. 특히 검색 유입이 있는 글을 덮어써버리면 트래픽과 신뢰도가 같이 흔들린다. 그래서 BloggerGent에서는 발행 기능을 개선할 때 SEO보다 먼저 “공개 글 덮어쓰기 금지”를 넣었다.

현재 구조에서는 이미 공개된 글에 다시 게시 버튼을 눌러 기존 포스트를 업데이트하는 흐름을 기본으로 두지 않는다. 일단 공개된 글은 보호하고, 새 글은 새 글로 생성하는 쪽을 기본값으로 잡는다. 여기에 중복 주제 차단까지 더해서 같은 주제가 반복 생성되는 것도 최대한 줄였다. 운영 도구라면 이런 가드레일이 반드시 필요하다.

SEO 메타 검증도 같은 맥락에서 중요하다. 많은 자동화 도구는 DB에 메타 설명이 저장돼 있으면 SEO가 끝난 것처럼 보인다. 하지만 Blogger는 그렇게 단순하지 않다. 실제 공개 페이지 `<head>`에 어떤 값이 들어가는지는 테마, 검색 설명 설정, 렌더링 방식에 따라 달라진다. 그래서 BloggerGent는 “저장값”과 “공개 페이지 상태”를 분리해서 보게 만들었다.

여기서 핵심은 `customMetaData`를 믿지 않는 것이다. 이 필드는 있어 보이지만, 실제 공개 페이지 메타 태그와 일치하지 않는 경우가 많다. 그래서 현재는 본문에 메타 설명을 심고, Blogger 테마 스크립트가 이를 읽어 `description`, `og:description`, `twitter:description`을 갱신하는 우회 구조를 사용한다. 완벽한 방식은 아니지만, Blogger 환경에서 현실적으로 통제 가능한 방법이다.

검증도 그 기준으로 바뀌었다. 이제 단순히 raw source만 보는 것이 아니라, 본문에 심긴 설명과 테마 패치 존재 여부까지 함께 본다. 즉 “API가 성공했으니 괜찮다”가 아니라 “실제 공개 페이지에서 기대한 SEO 설명이 동작하느냐”를 중심으로 판단한다.

정리하면 게시 안전장치와 SEO 검증은 별개의 기능이 아니다. 둘 다 공개 글을 잘 지키기 위한 장치다. 하나는 내용이 잘못 바뀌는 사고를 막고, 다른 하나는 검색 노출 상태를 점검한다. 운영 도구라면 두 기능이 같이 가야 한다.

## HTML 예시

```html
<h2>운영 안전장치</h2>
<ul>
  <li>공개 글 덮어쓰기 금지</li>
  <li>중복 주제 생성 차단</li>
  <li>수동 공개 게시 유지</li>
</ul>

<h2>SEO 검증 기준</h2>
<p>DB 저장값이 아니라 공개 페이지 기준으로 description, og:description, twitter:description 상태를 확인해야 한다.</p>
```
