# Step 5. AI 파이프라인을 공용 프롬프트가 아니라 블로그별 워크플로로 바꾼 이유

## 추천 제목

1. AI 블로그 자동화에서 공용 프롬프트를 버리고 블로그별 워크플로를 만든 이유
2. 여행 블로그와 미스터리 블로그를 같은 프롬프트로 돌리면 안 되는 이유
3. 주제 발굴, 글쓰기, 이미지 생성 단계를 어떻게 분리했는가

## 검색 설명

여행 블로그와 미스터리 블로그를 함께 운영하면서 공용 프롬프트의 한계를 느꼈고, 결국 블로그별 워크플로 구조로 바꾼 과정을 정리했다.

## 추천 태그

`#프롬프트엔지니어링 #AI파이프라인 #블로그자동화 #OpenAI #Gemini #워크플로설계`

## 본문

AI 자동화 프로젝트를 만들다 보면 처음에는 공용 프롬프트 하나로 모든 것을 처리하고 싶어진다. 실제로 주제 발굴, 글쓰기, 이미지 프롬프트 생성까지 한 번에 넣으면 속도도 빠르고 구현도 단순해 보인다. 하지만 여러 블로그를 같이 운영하기 시작하면 이 방식은 거의 반드시 무너진다.

예를 들어 한국 여행 블로그는 최신 행사 일정, 교통, 예산, 현장 팁처럼 실용 정보가 중요하다. 반면 미스터리 블로그는 사실과 전설의 경계, 사건 전개, 분위기, 이론 비교 같은 구조가 중요하다. 같은 모델을 써도 같은 프롬프트로는 둘 다 만족시키기 어렵다. 하나는 너무 건조해지고, 다른 하나는 너무 투박해진다.

그래서 BloggerGent에서는 “공용 템플릿”보다 “블로그별 워크플로”가 중심이 되도록 구조를 바꿨다. 즉 전역 프롬프트는 기본값이나 프리셋 역할만 하고, 실제 운영에서 수정하는 것은 블로그별 단계 프롬프트다. 이렇게 해야 여행 블로그는 밝고 실용적인 톤을 유지하고, 미스터리 블로그는 다큐멘터리 같은 흐름을 유지할 수 있다.

워크플로도 단순화했다. 기본 경로는 `주제 발굴 -> 글쓰기 패키지 -> 이미지 생성 -> HTML 조립 -> 게시 대기`다. 여기서 글쓰기 패키지는 제목, 메타 설명, 본문, FAQ, 이미지 프롬프트까지 포함한다. 필요하면 `이미지 프롬프트 정교화` 단계를 고급 옵션으로 추가할 수 있지만, 기본 사용자는 굳이 그 단계를 볼 필요가 없도록 했다.

이 구조의 장점은 두 가지다. 첫째, 사용자는 자신이 운영하는 블로그에 필요한 단계만 명확하게 본다. 둘째, 시스템 단계는 백엔드에서 유지하되 UI는 덜 복잡하게 만들 수 있다. 실제로 운영 도구는 기능이 많을수록 단계를 덜 보이게 정리하는 것이 중요하다.

또 모델 선택을 단계별로 두는 것도 큰 의미가 있다. 주제 발굴은 Gemini가 더 잘 맞을 수 있고, 글쓰기는 OpenAI가 더 안정적일 수 있다. 이미지 프롬프트는 또 다른 모델이 유리할 수 있다. 이런 선택지를 블로그별로 다르게 적용할 수 있어야 “서비스처럼 운영되는 자동화”에 가까워진다.

결국 AI 파이프라인의 핵심은 모델 숫자가 아니라, 어떤 블로그에 어떤 단계와 어떤 문체를 연결하느냐다. BloggerGent에서 가장 큰 변화도 바로 그 지점이었다.

## HTML 예시

```html
<h2>기본 워크플로</h2>
<p>주제 발굴 → 글쓰기 패키지 → 이미지 생성 → HTML 조립 → 게시 대기</p>

<h2>왜 블로그별로 분리했는가</h2>
<p>여행 블로그와 미스터리 블로그는 구조와 톤이 다르다. 공용 프롬프트보다 블로그별 워크플로가 결과 품질과 운영 안정성 모두에서 유리하다.</p>
```
