# Step 8. 지금까지 구현한 기능을 바탕으로 앞으로 확장할 방향

## 추천 제목

1. BloggerGent 구현 정리와 다음 단계: 자동화보다 중요한 운영 기준
2. 여러 Blogger 블로그를 운영하며 얻은 구조적 교훈 8가지
3. 블로그 자동화 프로젝트를 서비스처럼 다듬으며 배운 점

## 검색 설명

지금까지 구현한 BloggerGent의 구조를 정리하고, 앞으로 어떤 기능을 확장하면 서비스 완성도가 더 올라갈지 운영 관점에서 정리했다.

## 추천 태그

`#프로젝트회고 #사이드프로젝트운영 #Blogger서비스 #자동화도구 #개발회고 #운영전략`

## 본문

여기까지 구현하고 나니 처음 생각했던 프로젝트와 지금의 BloggerGent는 꽤 달라졌다. 시작은 “AI가 블로그 글을 잘 써주게 하자”에 가까웠지만, 실제로 남은 것은 “여러 개의 Blogger 채널을 안정적으로 운영하는 구조를 어떻게 만들 것인가”라는 질문이었다. 이 변화가 오히려 프로젝트를 더 현실적으로 만들었다고 생각한다.

가장 큰 교훈은 자동화의 핵심이 속도가 아니라 제어라는 점이다. 글을 빨리 쓰게 만드는 것은 생각보다 어렵지 않다. 하지만 블로그마다 다른 스타일을 유지하고, 공개 글을 덮어쓰지 않고, 생성과 검토와 게시를 분리하고, SEO 상태를 검증하는 것까지 고려하면 이야기가 완전히 달라진다. 결국 자동화는 사람을 대체하는 것이 아니라, 사람이 더 정확하게 운영할 수 있게 보조하는 방향이 더 현실적이었다.

두 번째 교훈은 “플랫폼 제약을 인정하는 설계”가 중요하다는 점이다. Blogger는 WordPress처럼 자유도가 높지 않다. 메타데이터도 API만 믿고 끝낼 수 없고, 테마 수준의 우회가 필요하다. 이런 상황에서는 이상적인 구조를 고집하기보다, 실제로 통제 가능한 범위를 기준으로 설계하는 편이 낫다. 이번에 메타 설명 처리도 결국 그런 방식으로 정리됐다.

세 번째 교훈은 운영 도구는 기록이 남아야 한다는 점이다. 어떤 프롬프트를 썼고, 어떤 모델을 선택했고, 어떤 글이 공개됐고, 어떤 글이 실패했는지가 다 남아야 이후 개선이 가능하다. 그래서 작업 수치, 글 목록, 공개 상태, 검증 상태를 대시보드에서 계속 볼 수 있게 한 것이 생각보다 큰 가치가 있었다.

앞으로 확장할 수 있는 방향도 비교적 명확하다. 첫째, 브라우저 자동화를 더 안정화하면 Blogger 편집기의 검색 설명 입력까지 후처리할 수 있다. 둘째, 블로그별 성과 데이터를 더 정교하게 묶으면 Search Console과 GA4 기반 운영 리포트가 가능해진다. 셋째, 프롬프트 편집 UI를 더 완성도 있게 만들면 다른 사람도 쉽게 자기 블로그 스타일에 맞게 재구성할 수 있다.

무엇보다 중요한 것은 이 프로젝트가 이제 단순 기능 모음이 아니라는 점이다. BloggerGent는 “생성-검토-게시-검증”이라는 운영 루프를 하나의 서비스 흐름으로 묶기 시작했다. 이 흐름이 안정될수록, 여행 블로그든 미스터리 블로그든 성격이 다른 채널을 한곳에서 운영하는 경험이 더 좋아질 것이다.

프로젝트를 길게 끌고 가려면 항상 질문을 다시 해야 한다. “이 기능이 멋있어 보이는가?”보다 “이 기능이 운영 사고를 줄이고, 실제 반복 작업을 편하게 만드는가?”가 더 중요하다. BloggerGent도 앞으로는 그 기준으로 조금씩 더 정리해 나갈 생각이다.

```html
<h2>다음 확장 방향</h2>

  <li>Search Console / GA4 기반 성과 리포트 강화</li>
  <li>블로그별 프롬프트 편집 UI 개선</li>
</ul>
```
