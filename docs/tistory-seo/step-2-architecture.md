# Step 2. BloggerGent 아키텍처를 어떻게 나눴는가

## 추천 제목

1. FastAPI + Next.js + Celery로 Blogger 자동화 아키텍처를 나눈 이유
2. 블로그 운영 도구는 왜 프론트, API, 워커를 분리해야 하는가
3. Blogger 자동화 서비스 구조 설계: 대시보드, API, 워커, DB를 분리한 이유

## 검색 설명

BloggerGent를 구현하며 프론트엔드, API, 워커, DB를 어떻게 나눴는지 설명한다. 단순 CRUD가 아니라 운영 도구 구조로 설계한 이유를 함께 정리했다.

## 추천 태그

`#FastAPI #Nextjs #Celery #PostgreSQL #Redis #서비스아키텍처 #블로그운영도구`

## 본문

프로젝트를 처음 만들 때는 백엔드 하나만 있어도 충분해 보인다. API에서 글을 만들고, 곧바로 이미지 생성과 발행까지 처리하면 되는 것처럼 느껴지기 때문이다. 하지만 실제로 기능이 늘어나고 블로그가 여러 개가 되면, 모든 일을 한 프로세스 안에서 처리하는 구조는 금방 유지보수가 어려워진다. 그래서 BloggerGent는 프론트엔드, API, 워커, 데이터 저장소를 명확하게 분리하는 방향으로 정리했다.

프론트엔드는 Next.js 기반 대시보드다. 여기서 중요한 역할은 단순히 버튼 몇 개를 눌러 API를 호출하는 것이 아니다. 어떤 블로그를 운영 중인지, 어떤 워크플로를 쓰는지, 생성된 글이 몇 개인지, 실패 작업은 무엇인지, SEO 메타 검증은 어떤 상태인지처럼 “운영자가 보고 판단해야 하는 정보”를 모아서 보여주는 것이 핵심이다. 그래서 프론트는 예쁜 화면보다도 작업 흐름을 명확하게 보여주는 데 초점을 맞췄다.

백엔드는 FastAPI를 사용했다. 이유는 명확하다. 설정, 블로그 가져오기, 글 생성, 발행, SEO 검증처럼 서로 다른 책임을 API 단위로 분리하기 좋고, 비동기 호출이나 외부 API 연동도 편하다. 무엇보다 문서화가 자연스럽기 때문에 대시보드와 분리해서 생각하기 쉽다. 실제로 운영 도구에서는 “무엇을 눌렀을 때 어떤 데이터가 오가는지”가 명확해야 디버깅이 쉽다.

워커는 Celery와 Redis를 붙였다. 블로그 운영에서 모든 요청이 즉시 끝나는 것은 아니다. 이미지 생성, 외부 API 호출, 예약 실행, 반복 작업은 웹 요청 하나 안에서 처리하면 불안정하다. 그래서 긴 작업은 워커로 넘기고, 대시보드는 작업 상태를 조회하는 쪽이 맞다고 판단했다. 이 구조 덕분에 나중에 스케줄러를 붙이거나, 실패 작업을 다시 돌리거나, 작업 로그를 정리하는 것도 훨씬 수월해졌다.

데이터 저장은 PostgreSQL을 중심으로 잡았다. 여기서 중요한 점은 글 자체만 저장하는 것이 아니라는 점이다. 블로그 설정, 워크플로 단계, 프롬프트 템플릿, 작업 상태, 생성된 글, 발행된 Blogger 포스트, 이미지 메타데이터, SEO 검증 결과 같은 운영 데이터를 분리해서 저장해야 한다. 이 구조가 있어야 “왜 이 글이 이렇게 만들어졌는지”를 추적할 수 있다.

이런 분리를 해두면 장점이 많다. 프론트는 운영 경험에 집중하고, API는 규칙과 검증에 집중하고, 워커는 오래 걸리는 작업을 처리하고, DB는 상태를 보존한다. 특히 공개 글 덮어쓰기 방지나 블로그별 프롬프트 분리 같은 기능은 이런 책임 분리가 되어 있어야 자연스럽게 구현된다.

결국 BloggerGent의 아키텍처는 화려한 기술 스택을 자랑하기 위한 구성이 아니다. “운영 안정성”을 얻기 위해 필요한 최소한의 분리다. 이게 있어야 여러 블로그를 한 대시보드에서 관리하면서도, 각 블로그의 개성을 유지하고 사고를 줄일 수 있다.
