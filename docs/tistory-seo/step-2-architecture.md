# Step 2. Blogger 자동화 시스템 아키텍처 설계 정리

## 추천 제목
- Blogger 자동화 시스템 아키텍처 설계: FastAPI, Next.js, Celery로 나눈 이유
- 구글 블로그 반자동화 구조 공개: 웹, API, 워커를 어떻게 나눴나
- Blogger SEO 자동화 아키텍처 설계: 프론트엔드부터 워커까지

## 검색 설명
Next.js, FastAPI, Celery, Redis, PostgreSQL을 사용해 Blogger 자동화 시스템을 어떻게 설계했는지 구조 중심으로 정리한 글입니다.

## 추천 태그
`FastAPI`, `Nextjs`, `Celery`, `Redis`, `PostgreSQL`, `시스템설계`

## 글 구조
- 왜 프론트엔드와 백엔드를 분리했는가
- 워커와 스케줄러를 따로 둔 이유
- 왜 PostgreSQL과 Redis 조합을 선택했는가
- 블로그별 워크플로 설계의 핵심

## 본문 샘플

이 프로젝트는 단순한 스크립트가 아니라 실제 운영형 서비스처럼 동작해야 했습니다. 그래서 화면은 Next.js, API는 FastAPI, 비동기 작업은 Celery, 상태 저장은 PostgreSQL, 큐는 Redis로 분리했습니다.

핵심은 **블로그마다 서로 다른 워크플로와 프롬프트를 가질 수 있어야 한다는 점**이었습니다. 여행 블로그와 미스터리 블로그는 같은 “글 생성” 기능을 써도 역할과 톤이 완전히 다르기 때문입니다.

## HTML 예시

```html
<h2>전체 구조를 어떻게 나눴나</h2>
<ul>
  <li><strong>web</strong>: 설정, 미리보기, 모니터링</li>
  <li><strong>api</strong>: 설정 저장, 블로그 import, 게시 요청</li>
  <li><strong>worker</strong>: AI 생성, 이미지 생성, HTML 조립</li>
  <li><strong>scheduler</strong>: 자동 주제 발굴 스케줄 실행</li>
</ul>

<h2>왜 워커를 따로 뒀나</h2>
<p>본문 생성과 이미지 생성은 요청 시간이 길고, 실패 재시도도 필요합니다.</p>
<p>그래서 사용자 화면 요청과 분리해야 안정적으로 운영할 수 있습니다.</p>
```

## 마무리 문장 예시

구조를 분리하고 나니 “한 번 돌고 끝나는 스크립트”가 아니라, 블로그 여러 개를 동시에 다룰 수 있는 운영형 서비스의 틀이 생겼습니다. 다음 글에서는 이 구조를 실제로 로컬에서 바로 실행되게 만드는 Docker 환경을 정리하겠습니다.
