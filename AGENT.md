# AGENT 운영 기준

## 1. 목적
이 문서는 Bloggent 저장소에서 "운영 코드"와 "리팩토링/실험 코드"를 분리해 관리하기 위한 기준이다.

## 2. 운영 브랜치 규칙
- 운영 브랜치는 `main` 단일 기준으로 관리한다.
- 단기 작업 브랜치는 완료 즉시 `main` 반영 후 삭제한다.
- 동일 트리 브랜치는 merge commit 없이 삭제 정리한다.

## 3. 운영 코드 정의
다음 경로는 운영 배포 대상이다.
- `apps/web`
- `apps/api/app`
- `apps/api/alembic`
- `prompts/channels`
- `infra`
- `wiki`

## 4. 비추적(커밋 금지) 원칙
- DB 데이터/덤프/백업/시크릿은 git에 포함하지 않는다.
- 로컬 실행 산출물(리포트/임시 HTML/캐시)은 추적하지 않는다.
- `.env`, `env/*`는 로컬 전용으로 유지한다.

## 5. 리팩토링/실험 코드 규칙
- 리팩토링/실험 파일은 반드시 `refactoring/` 아래에서만 작업한다.
- 운영 import 체인(`apps/api/app/*`, `apps/web/*`)에서 `refactoring/` 코드를 참조하지 않는다.
- 실험이 운영 반영될 때만 필요한 최소 코드만 운영 경로로 이동한다.

## 6. 에이전트(AI Agent) 협업 규칙
- 모든 기능 개발 및 리팩토링 시 `.cursorrules`(@rules)를 최우선으로 참조한다.
- 에이전트는 코드 수정 전 반드시 `REPO_STRUCTURE.md`를 통해 영향 범위를 파악한다.
- 운영 코드(`apps/`) 수정 시에는 `implementation_plan.md`를 통해 사용자 승인을 득하는 것을 원칙으로 한다.

## 7. 커밋 규칙
- 커밋 전에 `git status --short`로 범위를 확인한다.
- 의도하지 않은 생성물/비밀값이 포함되면 커밋하지 않는다.
- 문서/정책 변경은 코드 변경과 분리 커밋을 우선한다.
- 커밋 메시지는 한국어 또는 영어를 사용하되 명확하게 변경 사항을 기술한다.
